# Python builtins and their PEG encoding

This document lists all `py_*` builtins generated by the Python frontend
(`PyIR2Textual.ml`) and discusses how each is (or should be) handled
during PEG construction (`TextualPeg.ml`).

## PEG encoding categories

Each builtin falls into one of these categories:

- **Interpreted**: the PEG builder understands the semantics and produces
  specialized e-graph terms (e.g. locals map updates instead of opaque calls).
- **Pure opaque**: translated as a pure e-graph term `(builtin_name arg1 ... argN)`.
  No state threading. The term is structurally compared.
- **Effectful opaque**: translated as a pure term but threaded through
  `@seq(state, term)` to preserve execution order.
- **Skipped**: ignored during PEG construction (compiler artifacts).
- **Unsupported**: causes `L.die InternalError` — not yet handled.

## Locals management

These builtins interact with the mutable `*PyLocals` frame. The PEG builder
interprets them to build proper SSA with `@phi` at join points.

| Builtin | PEG encoding | Category |
|---------|-------------|----------|
| `py_store_fast("x", locals, val)` | Updates locals map: `locals["x"] <- convert(val)`. No e-graph term. | Interpreted |
| `py_load_fast("x", locals)` | Reads from locals map: returns `locals["x"]`. No e-graph term. | Interpreted |
| `py_nullify_locals(locals, ...)` | Skipped — compiler artifact that clears variables. | Skipped |
| `py_store_name("x", globals, val)` | Effectful opaque — writes to module namespace. | Effectful opaque |
| `py_load_name("x", globals)` | Effectful opaque — reads from module namespace. | Effectful opaque |
| `py_store_global("x", globals, val)` | Effectful opaque. | Effectful opaque |
| `py_load_global("x", globals)` | Effectful opaque. | Effectful opaque |
| `py_store_deref(...)` | Effectful opaque — writes to closure cell. | Effectful opaque |
| `py_load_deref(...)` | Effectful opaque — reads from closure cell. | Effectful opaque |
| `py_load_closure(...)` | Effectful opaque. | Effectful opaque |
| `py_load_class_deref(...)` | Effectful opaque. | Effectful opaque |
| `py_load_fast_check(...)` | Effectful opaque (like `load_fast` but may raise). | Effectful opaque |
| `py_load_fast_and_clear(...)` | Effectful opaque. | Effectful opaque |
| `py_load_locals(...)` | Effectful opaque. | Effectful opaque |
| `py_load_from_dict_or_deref(...)` | Effectful opaque. | Effectful opaque |
| `py_delete_fast(...)` | Effectful opaque. | Effectful opaque |
| `py_delete_name(...)` | Effectful opaque. | Effectful opaque |
| `py_delete_global(...)` | Effectful opaque. | Effectful opaque |
| `py_delete_deref(...)` | Effectful opaque. | Effectful opaque |

### Discussion

Only `py_store_fast` / `py_load_fast` are interpreted today. This is
sufficient for Python Textual v0 where all local variable access goes
through `*PyLocals`. The other variants (`_name`, `_global`, `_deref`)
are treated as opaque effectful calls, which is conservative but correct.

Future work: `py_store_name` / `py_load_name` could potentially be
interpreted similarly for module-level code.

## Constants and literals

These create Python objects from literal values. They are **pure** — no
side effects.

| Builtin | PEG encoding | Notes |
|---------|-------------|-------|
| `py_make_none()` | Pure opaque: `$builtins.py_make_none` | Generated at every function entry |
| `py_make_int(n)` | Pure opaque: `($builtins.py_make_int n)` | |
| `py_make_float(f)` | Pure opaque: `($builtins.py_make_float f)` | |
| `py_make_string(s)` | Pure opaque: `($builtins.py_make_string s)` | |
| `py_make_complex(r, i)` | Pure opaque | |
| `py_make_bytes(b)` | Pure opaque | |
| `py_bool_true` | Pure opaque | |
| `py_bool_false` | Pure opaque | |
| `py_invalid_unicode(...)` | Pure opaque | |

### Discussion

`py_make_none` is always generated at function entry (the frontend creates
`n0 = $builtins.py_make_none()` for every function). This is harmless but
pollutes the `@seq` state chain since the current PEG treats all `Call`
expressions as effectful. A possible improvement: recognize `py_make_*`
builtins as pure and skip state threading for them.

### Comparison operators

| Builtin | Python op |
|---------|-----------|
| `py_compare_lt` | `<` |
| `py_compare_le` | `<=` |
| `py_compare_eq` | `==` |
| `py_compare_neq` | `!=` |
| `py_compare_gt` | `>` |
| `py_compare_ge` | `>=` |
| `py_compare_in` | `in` |
| `py_compare_not_in` | `not in` |
| `py_compare_is` | `is` |
| `py_compare_is_not` | `is not` |
| `py_compare_exception` | exception matching |

### Discussion

Currently all these operators are treated as **effectful opaque** (threaded
through `@seq`) because the PEG builder treats all `Call` expressions as
side-effectful. This is overly conservative — arithmetic and comparisons
are pure.

**Proposed improvements:**

1. **Mark pure builtins**: recognize `py_binary_*`, `py_unary_*`,
   `py_compare_*`, `py_make_*` as pure — do not thread through `@seq`.
   This would allow reordering of pure computations.

2. **Commutativity rules**: add rewrite rules for commutative operators:
   ```
   ($builtins.py_binary_add ?X ?Y) ==> ($builtins.py_binary_add ?Y ?X)
   ($builtins.py_binary_multiply ?X ?Y) ==> ($builtins.py_binary_multiply ?Y ?X)
   ($builtins.py_binary_and ?X ?Y) ==> ($builtins.py_binary_and ?Y ?X)
   ($builtins.py_binary_or ?X ?Y) ==> ($builtins.py_binary_or ?Y ?X)
   ($builtins.py_binary_xor ?X ?Y) ==> ($builtins.py_binary_xor ?Y ?X)
   ```

3. **In-place equivalence**: `py_inplace_add(x, y)` is semantically
   equivalent to `py_binary_add(x, y)` for immutable types (int, float,
   str). This could be an optional rule but is unsound for mutable types
   (list, etc.).

## Control flow

| Builtin | PEG encoding | Notes |
|---------|-------------|-------|
| `py_bool(x)` | Effectful opaque | Truthiness test, generated at `if` terminators |

### Discussion

`py_bool` is generated for every `if` condition: `if $builtins.py_bool(n3)`.
It appears in the PEG as the condition of `@phi` nodes:
`@phi(($builtins.py_bool @param:c), then_result, else_result)`.

It is currently threaded through `@seq` as effectful, but `py_bool` is
actually pure (it only reads, never mutates). Marking it pure would
simplify PEG terms.

## Priority improvements for PEG accuracy

1. **Mark pure builtins** — `py_make_*`, `py_binary_*`, `py_unary_*`,
   `py_compare_*`, `py_bool`, `py_build_tuple`, `py_build_frozen_set`
   should not be threaded through `@seq`. This is the single biggest
   improvement for PEG precision.

2. **Commutativity rules** — for `py_binary_add`, `py_binary_multiply`,
   `py_binary_and`, `py_binary_or`, `py_binary_xor`.

3. **Interpret `py_store_name`/`py_load_name`** — like `store_fast`/`load_fast`
   but for module-level code.

4. **`py_bool` in conditions** — could be absorbed into the `@phi` condition
   rather than being a separate opaque call.
